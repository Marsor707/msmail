# 微软邮箱管理系统 改版目标清单

## 1. 改版目标

这次改版的核心目标不是增加新功能，而是把现有功能做成一个更专业、更清晰、更有产品感的后台界面。

希望设计重构后达到以下效果：

- 让首页一眼看出“这是一个邮箱管理工作台”，而不是普通的技术 Demo 页面
- 让账号管理、邮件列表、邮件详情三类页面的层级关系更清楚
- 让高频操作更顺手，减少“找按钮”“找信息”的成本
- 让页面既适合内部使用，也具备可以对外演示的视觉完成度

---

## 2. 本次改版不变的部分

以下内容默认不改，设计需要在这些约束内进行：

- 功能范围不变
- 页面数量不变
- 主要业务流程不变
- 仍然保留三个核心页面
  - 首页 / 账号管理页
  - 邮件列表页
  - 邮件详情页
- 仍然保留当前已有功能
  - 批量导入账号
  - 删除账号
  - 查看邮件列表
  - 查看邮件详情
  - 对外 API 能力

---

## 3. 希望解决的核心问题

### 3.1 首页问题

- 当前首页更像“左右两个卡片拼在一起”，缺少真正的后台首页结构
- 导入区和账号列表区视觉权重接近，没有主次
- 缺少更强的导航感和工作台感

### 3.2 邮件列表页问题

- 当前列表页信息量偏少，像一页简单的记录列表
- 没有形成“筛选 + 浏览 + 快速判断”的阅读节奏
- 列表项之间的层级关系还不够明显

### 3.3 邮件详情页问题

- 当前详情页偏“静态展示”，缺少操作区和阅读引导
- 元信息卡片区与正文区之间的节奏感还不够强
- 整体更像技术预览页，而不是完整工作页面

---

## 4. 设计改版后的目标状态

### 4.1 整体气质

希望新版界面更接近以下关键词：

- 专业
- 清爽
- 稳定
- 现代
- 有一定数据后台气质
- 但不能太硬、太像传统管理系统

不希望出现的问题：

- 过度炫技
- 太像模板站
- 太像普通表单系统
- 设计语言过于平淡，没有识别度

### 4.2 信息架构

希望设计后做到：

- 首页重点突出“账号管理”和“导入能力”
- 邮件列表页重点突出“筛选效率”和“快速浏览”
- 邮件详情页重点突出“阅读体验”和“邮件元信息”

### 4.3 交互体验

希望设计师优先提升这些体验：

- 高优先级操作更容易被找到
- 页面状态更明确
- 空态、异常态、成功态更完整
- 用户在不同页面之间切换时有统一的导航规则

---

## 5. 页面级改版要求

## 5.1 首页 / 账号管理页

首页建议重做成“工作台”型页面，而不是简单左右分栏。

希望首页重点体现：

- 系统定位
- 核心统计
- 批量导入
- 账号管理列表

建议重点优化：

- 首页顶部区域的品牌感
- 统计信息的表达方式
- 导入流程的可理解性
- 账号列表的可浏览性
- 删除、查看邮件等动作的操作感

希望首页最终给人的感觉是：

- 打开就知道能做什么
- 重点区域一眼能定位
- 不需要解释也能开始操作

## 5.2 邮件列表页

邮件列表页建议从“单列卡片列表”升级为更完整的邮件浏览工作区。

建议设计重点考虑：

- 邮件列表的层级
- 发件人、主题、时间、状态的优先级
- 未读邮件的识别方式
- 附件邮件的识别方式
- 选择最近邮件数量的控件形式
- 是否需要更强的列表头部操作区

如果设计师有更成熟的邮件产品经验，可以探索：

- 左侧列表 + 右侧预览
- 或者更像邮箱客户端的双栏结构

前提是：

- 不增加复杂功能
- 只优化现有浏览体验

## 5.3 邮件详情页

邮件详情页建议突出“阅读”和“信息结构”。

设计重点建议：

- 邮件标题区更有层级
- 发件人区更清楚
- 元信息区更规整
- 正文区更适合长文阅读
- 页面可考虑增加固定的操作位

操作位可以预留但不一定开发：

- 返回列表
- 上一封 / 下一封
- 复制 Message ID
- 附件区域预留

---

## 6. 状态设计要求

本次改版希望设计师补足以下状态，而不是只画“正常态”：

- 空状态
  - 没有导入邮箱
  - 没有邮件

- 加载状态
  - 导入中
  - 刷新中
  - 邮件详情加载中

- 成功状态
  - 导入成功
  - 刷新成功

- 错误状态
  - Token 失效
  - 接口失败
  - 邮件读取失败

设计稿里至少应体现：

- 主要正常态
- 一个空态
- 一个错误态

---

## 7. 视觉与品牌建议

建议设计师在以下方向上做增强：

- 系统名称的视觉识别
- 页面头部统一风格
- 按钮层级更鲜明
- 卡片结构更有节奏
- 页面之间保持统一但不要雷同

建议保留的优点：

- 浅色系、清爽
- 大圆角、柔和
- 不要做成厚重的传统后台

建议增强的方向：

- 更明确的品牌主色
- 更成熟的字体层级
- 更专业的后台布局
- 更明显的状态色和重点色

---

## 8. 响应式要求

虽然这个系统主要偏桌面使用，但新版设计仍需要考虑移动端和窄屏场景。

最低要求：

- 首页在窄屏下不会布局错乱
- 列表页在窄屏下仍然能浏览邮件
- 详情页正文在窄屏下仍然可读

如果设计师输出多端稿，优先级建议：

- 桌面端优先
- 窄屏适配次之

---

## 9. 本次设计交付建议

建议设计师至少交付以下内容：

- 首页高保真稿
- 邮件列表页高保真稿
- 邮件详情页高保真稿
- 一个空态示例
- 一个错误态示例
- 基础色板、字体层级、按钮规范

如果时间允许，建议额外交付：

- 组件规范摘要
- 列表卡片规范
- 表格 / 卡片两种账号展示方案对比

---

## 10. 设计优先级

如果设计时间有限，优先级建议如下：

1. 首页信息架构重做
2. 邮件列表页体验重做
3. 邮件详情页阅读体验提升
4. 空态 / 错误态补齐
5. 统一组件样式规范

---

## 11. 给设计师的一句话总结

这次不是做“更花”的页面，而是把现在已经能用的微软邮箱管理系统，重做成一个更像正式产品的内部工作台。
